# Implementation Plan: Database Integration

**Branch**: `004-db-integration` | **Date**: 2026-05-05 | **Spec**: [spec.md](./spec.md)  
**Input**: Feature specification from `specs/004-db-integration/spec.md`

## Summary

Install Prisma 7 fresh, configure it for Neon PostgreSQL using the pooled + direct URL split, define the full application schema (User, Item, ItemType, Collection, Tag, ItemTag) merged with NextAuth adapter models (Account, Session, VerificationToken), create and run the initial migration against the Neon `develop` branch, expose a hot-reload-safe Prisma singleton at `src/lib/db.ts`, and seed the 7 system ItemTypes.

## Technical Context

**Language/Version**: TypeScript 5 (strict mode), Node.js 20.19+  
**Primary Dependencies**: `prisma@7`, `@prisma/client@7`, `@prisma/adapter-pg`, `tsx` (dev)  
**Storage**: Neon PostgreSQL (`develop` branch) — pooled URL for runtime, direct URL for migrations  
**Testing**: `pnpm build` — no unit tests in this feature  
**Target Platform**: Vercel (Next.js 16 App Router)  
**Project Type**: web-service  
**Performance Goals**: Migration completes in under 10 seconds; connection established in under 5 minutes from clone  
**Constraints**: Prisma 7 (breaking changes from v6); `db push` prohibited; pooled URL required for Vercel serverless  
**Scale/Scope**: Single-user dev setup now; schema designed for multi-user production

## Implementation Notes

### .gitignore
- Add `!.env.example` exception so the template file is committed despite `.env*` rule
- Add `src/generated/prisma/` to ignore the generated Prisma client

### package.json
- Add `"postinstall": "prisma generate"` — runs automatically after `pnpm install` locally and on Vercel
- Add `"engines": { "node": ">=20.19" }` — Prisma 7 minimum requirement
- Production Vercel build command: `prisma migrate deploy && next build`

### Dependencies
- Production: `prisma@7`, `@prisma/client@7`, `@prisma/adapter-pg`, `pg`
- Dev: `tsx`, `@types/pg`

### prisma.config.ts (Prisma 7 specific)
- Use `import { config } from "dotenv"; config({ path: ".env.local" })` — NOT `import "dotenv/config"` (which loads `.env` not `.env.local`)
- `datasource.url` uses `env("DIRECT_URL")` (unpooled) — Prisma CLI needs direct connection for DDL
- `migrations.seed` set to `"tsx prisma/seed.ts"` — replaces the v6 `package.json prisma.seed` field
- No `engine` property — removed in Prisma 7

### schema.prisma (Prisma 7 specific)
- `generator client` uses `provider = "prisma-client"` (not `"prisma-client-js"`)
- `output = "../src/generated/prisma"` — client generated outside `node_modules`
- `datasource db` has no `url` field — handled by `prisma.config.ts`
- `User.email` is `String @unique` (non-nullable) — DevStash requires email for all auth flows

### src/lib/db.ts
- Import from `@/generated/prisma/client` (resolves via existing `@/*` → `src/*` alias)
- `PrismaPg` adapter initialized with pooled `DATABASE_URL`
- `globalThis` guard prevents connection pool exhaustion on Next.js hot-reload

### prisma/seed.ts
- System ItemTypes use hardcoded stable IDs (e.g., `"itemtype_snippet"`) as upsert key
- `upsert` on `id` — safe to run multiple times, rename-safe
- No full-text search indexes — deferred to the search feature

### Merge of User model
- Application fields: `password?`, `isPro`, `stripeCustomerId?`, `stripeSubscriptionId?`
- NextAuth required additions: `name?`, `emailVerified?`, `image?`
- `email` stays non-nullable (`String @unique`) — deviates from NextAuth adapter docs intentionally
- Relations added: `accounts Account[]`, `sessions Session[]`
